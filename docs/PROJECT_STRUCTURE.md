### 1.1 `frontend/`

**Purpose**: The single-page application вЂ” browser entry point for all users.

**What belongs inside**: React components, pages, hooks, API client code, styles, static assets, Vite configuration, ESLint/Prettier config, frontend-specific tests.

**What should never be placed inside**: Server-side code, database access, API route definitions, Python files, secret keys, backend configuration.

---

### 1.2 `backend/`

**Purpose**: The FastAPI HTTP server вЂ” handles API requests, authentication, and orchestrates the analysis pipeline.

**What belongs inside**: Route handlers, business logic services, database models and migrations, Pydantic schemas, application configuration, authentication logic, exception handlers, middleware.

**What should never be placed inside**: AI agent implementations (those live in `backend/agents/`), Celery task definitions (those live in `workers/`), frontend assets, database seeds (those live in `scripts/`).

---

### 1.3 `workers/`

**Purpose**: Celery worker processes that execute the AI analysis pipeline asynchronously. A separate deployment unit from the backend API.

**What belongs inside**: Celery app configuration, task definitions, agent orchestration logic, task routing configuration, worker-specific logging setup.

**What should never be placed inside**: HTTP route handlers, database migrations, frontend code, shared library code (use `shared/` for that).

---

### 1.4 `shared/`

**Purpose**: Python code shared between `backend/` and `workers/`. Avoids duplication of schemas, constants, and utilities.

**What belongs inside**: Pydantic models used by both API and workers, enum definitions, constant values (limits, defaults, error codes), utility functions that have zero HTTP or infrastructure dependency.

**What should never be placed inside**: Business logic, database queries, AI agent code, HTTP-aware code, infrastructure-specific configuration, secrets.

---

### 1.5 `docs/`

**Purpose**: All project documentation вЂ” the source of truth for architecture, design decisions, and operational procedures.

**What belongs inside**: ADRs, runbooks, API documentation, architecture diagrams, database schemas, onboarding guides, contribution guidelines, changelog.

**What should never be placed inside**: Application code, configuration files, secrets, build artifacts, generated documentation that lives in the codebase's own repo (like API docs generated by FastAPI).

---

### 1.6 `docker/`

**Purpose**: Dockerfiles and Docker Compose configurations for local development, CI, and production deployment.

**What belongs inside**: `Dockerfile` per service (api, worker, frontend), `docker-compose.yml` for local dev, `docker-compose.prod.yml` for production, `.dockerignore` files, Docker entrypoint scripts.

**What should never be placed inside**: Application code, secrets, built images, Kubernetes manifests (use `scripts/deploy/` or a separate ops repo).

---

### 1.7 `scripts/`

**Purpose**: Maintainable, reusable scripts for development workflow, CI/CD, database operations, and maintenance tasks.

**What belongs inside**: Database seed scripts, migration helper scripts, CI/CD pipeline scripts, setup/teardown scripts, data anonymization scripts, backup/restore scripts, one-time data migration scripts.

**What should never be placed inside**: Application code, long-running daemons, configuration that belongs in Docker Compose, secrets.

---

### 1.8 `tests/`

**Purpose**: Top-level integration, end-to-end, and cross-service tests that span multiple components.

**What belongs inside**: E2E tests (frontend в†’ API в†’ worker в†’ database), cross-service integration tests, performance/load tests, test fixtures shared across service boundaries, test configuration.

**What should never be placed inside**: Unit tests for a single service (those live in that service's own `tests/` directory), AI agent fixture data (those live in `backend/tests/fixtures/`), environment secrets.

---

### 2.1 Backend Folder Explanations

---

#### `backend/app/api/`

**Purpose**: The HTTP layer вЂ” route handlers, request validation, response serialization.

**What belongs inside**: FastAPI route functions, endpoint-level dependencies (auth, pagination), exception-to-HTTP mapping, router aggregation.

**What should never be placed inside**: Business logic, database queries, AI agent calls, import of Celery tasks directly (call services instead).

---

#### `backend/app/services/`

**Purpose**: Business logic orchestration вЂ” coordinates between API layer, repositories, and external dependencies.

**What belongs inside**: Use-case-level logic, transaction coordination, cross-entity operations, service-to-service communication.

**What should never be placed inside**: HTTP awareness (request/response objects), direct database queries (use repositories), AI agent logic.

---

#### `backend/app/repositories/`

**Purpose**: Data access layer (DAO pattern) вЂ” encapsulates all database queries behind a consistent interface.

**What belongs inside**: SQLAlchemy query methods (`get`, `get_multi`, `create`, `update`, `delete`), pagination logic, query filtering, raw SQL when necessary for performance.

**What should never be placed inside**: Business logic, transaction management (that belongs in services), HTTP awareness, AI logic, Pydantic schema definitions.

---

#### `backend/app/ai/`

**Purpose**: All AI-related functionality grouped under one subtree вЂ” agents, orchestration, retrieval, embeddings, prompts, and evaluation.

**What belongs inside**: Agent implementations extending `BaseAgent`, LangGraph pipeline orchestration, file chunking and context assembly, FAISS vector embeddings and search, prompt templates and registry, confidence scoring and output validation.

**What should never be placed inside**: HTTP routing, database queries (use repositories via services), report formatting, business logic that is not AI-specific.

---

##### `backend/app/ai/agents/`

**Purpose**: Individual AI agents вЂ” each self-contained unit that produces structured findings from repository data.

**What belongs inside**: Agent classes extending `BaseAgent`, agent-specific output parsers, agent configuration.

**What should never be placed inside**: HTTP routing, database repository calls (use services), report aggregation logic, prompt template text files (those go in `ai/prompts/templates/`).

---

##### `backend/app/ai/orchestration/`

**Purpose**: LangGraph-based pipeline orchestration вЂ” defines the graph of agent execution with sequential, parallel, and conditional branches.

**What belongs inside**: LangGraph state definitions, node functions, edge routing logic, graph compilation.

**What should never be placed inside**: Celery task definitions, agent implementations, database access, prompt templates.

---

##### `backend/app/ai/retrieval/`

**Purpose**: Context retrieval pipeline вЂ” transforms repository files into LLM-consumable context.

**What belongs inside**: File chunking strategies, context window assembly and truncation, file loading and language detection.

**What should never be placed inside**: Embedding computation (goes in `embeddings/`), agent prompt logic, HTTP awareness, database model definitions.

---

##### `backend/app/ai/embeddings/`

**What belongs inside**: One `PageName.tsx` per route, page-specific CSS module, page-specific components and hooks co-located in the page folder.

**Purpose**: Reusable, presentational UI components вЂ” the design system primitives.

**What belongs inside**: Generic components that accept props and render UI. Each component in its own folder with component file, CSS module, and test file.

**What should never be placed inside**: Page-specific components (those stay in the page folder), business logic, API calls, React Query hooks, routing logic.

#### `frontend/src/hooks/`

**Purpose**: Shared custom React hooks вЂ” reusable stateful logic.

**Purpose**: Typed API client layer вЂ” all external communication.

**What belongs inside**: Axios/fetch instance with auth interceptor, per-resource API function modules, error transformation logic, token refresh handling.

**What should never be placed inside**: React hooks, UI components, React Query `useQuery` calls, routing logic, state management.

**Purpose**: Shared TypeScript type definitions.

**What belongs inside**: API response/request interfaces, domain model interfaces, UI-related types, auth types, generic utility types.

**Purpose**: Static binary assets bundled with the application.

**What belongs inside**: SVG icons, PNG/JPG images, illustration files, self-hosted web fonts.

#### `frontend/src/styles/`

## 10. File Organization Rules

1. **One component, one folder**. Every React component or Python class with its own module gets its own folder when it has related assets (CSS module, test file, sub-components). Leaf utilities with no related assets can be a single file.

2. **Co-locate tests with source**. Unit tests live next to the module they test. Integration tests live in the project-level `tests/` directory. E2E tests live in the root `tests/` directory.
   - Frontend: `src/components/Button/Button.test.tsx`
   - Backend: `app/services/test_analysis_service.py` в†’ `tests/unit/services/test_analysis_service.py`

3. **One export per module**. A module (file) exports one primary thing вЂ” a class, a function, or a component. Secondary exports are rare and intentional.

4. **Flat is better than nested**. Do not nest beyond 4 levels from the service root (`backend/app/`). Agents can go deeper by agent name but should not add intermediate categorization folders.

5. **No barrel files**. Explicit imports over `__init__.py` re-exports. This keeps import chains traceable and avoids circular imports.

| File type | Soft limit | Hard limit | Action at soft limit |

| Test file | 200 lines | 400 lines | Split test cases into multiple files |
| Route handler (Python) | 100 lines | 200 lines | Extract logic to services |

Enforcement: Code review gates. CI will warn at soft limit (comment in PR) and fail at hard limit.

**Rule**: No folder path (from service root) should exceed 5 segments.

- Workers service root: `workers/worker/` вЂ” max depth `workers/worker/tasks/` (3 segments)

Enforcement: CI check using a script that walks directories and flags violations.

## 13. Import Rules

### 13.1 Backend (`backend/app/`)

```
Import order (enforced by Ruff):
1. Standard library     (os, json, uuid, datetime)
2. Third-party          (fastapi, sqlalchemy, pydantic, celery)
3. Shared               (from shared import ...)
4. Local                (from app.api import ..., from app.services import ...)

Rules:
- No importing across sibling service boundaries (e.g., services/ importing from ai/agents/)
  Use the service layer as the intermediary.
- Repository layer imports only: db/, schemas/, core/exceptions/
- Services import from: repositories/, schemas/, core/
- API layer imports from: services/, schemas/, core/, deps.py
- AI agents import from: ai/retrieval/, ai/embeddings/, ai/prompts/, parser/, security/, shared/schemas/
- AI orchestration imports from: ai/agents/, shared/schemas/
- Report builders import from: ai/agents/ (finding schemas), shared/schemas/
- Parser modules import only from: parser/base.py (parser interface), shared/
- No circular imports: api/ в†’ services/ в†’ repositories/ в†’ db/ (one direction only)
- Type-only imports: use `from typing import TYPE_CHECKING` + `if TYPE_CHECKING:` block
```

### 13.2 Workers (`workers/worker/`)

```
Import order (same as backend):
1. Standard library
2. Third-party (celery, langgraph)
3. Shared (from shared import ...)
4. Local (from worker import ...)

Rules:
- Tasks import from: backend services, shared/, backend/app/ai/orchestration/
- Tasks never import agents directly вЂ” they invoke orchestration
- Workers never import from: backend/app/api/
```

### 13.3 Frontend (`frontend/src/`)

```
Import order (enforced by ESLint import plugin):
1. Node built-ins        (react, react-dom)
2. Third-party           (@tanstack/react-query, react-router-dom)
3. Internal absolute     (@/types, @/hooks, @/services)
4. Internal relative     (./components, ./hooks)

Path aliases (via vite.config.ts):
  @/  в†’ src/
  @/pages в†’ src/pages/
  @/components в†’ src/components/
  @/hooks в†’ src/hooks/
  @/services в†’ src/services/
  @/types в†’ src/types/
  @/assets в†’ src/assets/
  @/styles в†’ src/styles/

Rules:
- Pages import from: components/, hooks/, services/, types/, styles/
- Components import from: components/ (sub-components), types/, styles/
- Hooks import from: services/, types/
- Services import from: types/, api.ts
- No importing from pages/ (pages are leaf nodes)
- No importing from assets/ into TypeScript logic (use imports in HTML/CSS)
- Absolute imports preferred for cross-module imports. Relative imports only for parent-sibling within the same feature module.
- CSS Modules import only from: styles/tokens.css (via CSS @import or `composes:`)
```

### 13.4 Shared (`shared/`)

```
Import order:
1. Standard library
2. Third-party (pydantic)

Rules:
- Shared imports from: nothing in app/ or worker/
- Shared is a zero-dependency layer (except pydantic for schemas)
- Backend and workers import from shared, not the other way around
```

## 14. Adding a New Agent вЂ” Extension Path

To add a new agent (ensuring no major refactoring is needed):

1. Add agent class in `backend/app/ai/agents/new_agent.py` extending `BaseAgent`
2. Add agent output schema in `shared/schemas/agent.py`
3. Add agent prompt template in `backend/app/ai/prompts/templates/new_agent.txt`
4. Register agent in pipeline graph in `backend/app/ai/orchestration/graph.py` (LangGraph node)
5. Add report section builder in `backend/app/reports/builders/new_agent.py`
6. Add agent name to planner configuration in `shared/constants/enums.py`
7. Write agent tests in `backend/tests/unit/ai/agents/`

No changes needed to: API routes, database models, repository layer, frontend pages, deployment config.

## 15. Rules of Thumb

- **If a folder has only one file, it doesn't need to be a folder.** Flatten it.
- **If a file exceeds 400 lines, split it.** Your future self will thank you.
- **If you're tempted to create a `utils/` subfolder, think twice.** Named folders communicate intent. `utils/` communicates "I didn't know where to put this."
- **Prefer duplication over premature abstraction.** Three occurrences of similar code is a pattern; two is coincidence.
- **Imports should tell a story.** Reading the imports of a file should reveal what that file depends on. If imports are too many, the file is doing too much.
- **CSS Modules belong with their component.** Not in a global styles folder. Global styles are only for reset, tokens, and animations.
