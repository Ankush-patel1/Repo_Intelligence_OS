# Repo Intelligence OS — Database Design

## Conventions

- All tables use `UUID` primary keys (generated server-side).
- All tables include `created_at` and `updated_at` timestamps.
- Soft deletes where applicable (`deleted_at` column).
- Foreign keys are indexed.
- JSON columns use `postgresql.JSONB`.

---

## Table: `reports`

Complete analysis reports. The `sections` JSONB column holds all report sections. Individual sections are also extractable.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| id | UUID | PK | |
| analysis_id | UUID | FK → analyses.id, UNIQUE, NOT NULL | |
| user_id | UUID | FK → users.id, NOT NULL | For access control |
| repository_id | UUID | FK → repositories.id, NOT NULL | |
| version | INTEGER | NOT NULL, DEFAULT 1 | Incremented on re-analysis |
| title | VARCHAR(255) | NOT NULL | Auto-generated |
| summary | TEXT | NOT NULL | Executive summary |
| sections | JSONB | NOT NULL | All report sections keyed by slug |
| scores | JSONB | NOT NULL, DEFAULT '{}' | Quality scores per section |
| tags | TEXT[] | NULLABLE | Auto-tags / user tags |
| is_public | BOOLEAN | NOT NULL, DEFAULT FALSE | Future: shareable link |
| created_at | TIMESTAMPTZ | NOT NULL | |
| updated_at | TIMESTAMPTZ | NOT NULL | |
| deleted_at | TIMESTAMPTZ | NULLABLE | |

**Indexes**:
- `idx_reports_analysis_id` on `analysis_id` (UNIQUE)
- `idx_reports_user_id` on `user_id`
- `idx_reports_repository_id` on `repository_id`

**`sections` JSONB structure**:

```json
{
  "executive_summary": { ... },
  "repository_overview": { ... },
  "architecture_analysis": { ... },
  "feature_inventory": { ... },
  "security_audit": { ... },

```

## Entity Relationships

```
users 1───* repositories
users 1───* analyses
users 1───* reports

repositories 1───* analyses

analyses 1───* findings
analyses 1───* file_index
analyses 1───* analysis_errors
analyses 1───1 reports

reports 1───* roadmap_items
reports.sections ──▶ JSON (denormalized section data)

findings 1───* roadmap_items (through finding_id)
```

## Migration Strategy

- Alembic for schema migrations.
- Naming convention for constraints: `{table}_{column}_fkey`, `{table}_pkey`, `{table}_{column}_idx`.
- All new tables are created in the `public` schema (MVP).
- JSONB columns do not have enforced schemas — validated at the application layer via Pydantic.
